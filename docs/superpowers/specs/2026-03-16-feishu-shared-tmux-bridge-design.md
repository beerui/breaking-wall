# 飞书接入共享 tmux 会话桥接（设计稿）

**日期：** 2026-03-16  
**目标：** 让飞书接入你正在使用的 `cc/cx` 持续对话模式，但底层不再尝试接管现有 Windows PowerShell，而是统一接入 WSL 内的共享 `tmux` 会话。

---

## 1. 背景与目标

### 1.1 要解决的问题
- 你不在电脑前时，仍然希望通过飞书继续推进 `cc/cx` 的工作。
- 本地终端和飞书必须共享同一条会话上下文，而不是各自启动新会话。
- 你能接受调整日常使用方式，但不希望改变“持续对话”的核心体验。

### 1.2 核心结论
- **不再尝试接管已经运行中的 Windows PowerShell/Console 会话。**
- **统一把工作会话放进 WSL 内的 `tmux` session。**
- 本地终端和飞书都连到这条共享会话。

### 1.3 非目标
- 不支持半路附着到任意一个已存在的 Windows Console 窗口。
- 不支持 GUI 远控。
- 首期不做多会话管理面板、不做会话创建/删除/重命名。

---

## 2. 用户体验

### 2.1 本地使用方式
- 你在 WSL 中进入固定的 `tmux` session。
- 在该 session 内运行 `cc` 或 `cx`。
- 继续像现在一样在终端里与工具交互。

### 2.2 飞书使用方式
- 普通文本：发送到当前绑定的共享会话。
- `/cc` `/cx`：切换飞书当前绑定目标。
- `/status`：查看当前绑定工具与共享会话状态。
- `/reset`：重新绑定共享会话，不再表示“重启 PTY”。

### 2.3 关键体验约束
- 飞书不是新开一个 AI 会话，而是把输入写入同一个共享终端。
- 你在本地发过的内容，飞书能接着继续。
- 你在飞书发过的内容，回到本地终端也能继续看到同一上下文。

---

## 3. 架构方案

### 3.1 推荐方案
- 主方案：Windows 本机运行 Agent。
- 共享会话运行在 WSL。
- 会话载体是 `tmux`。
- `cc` / `cx` 在 `tmux` pane 中运行。
- Agent 通过 `wsl.exe` 调用 `tmux` 完成输入注入与输出抓取。

### 3.2 后续扩展
- 同一套接口后续可支持远端 Linux 服务器。
- 但首期实现优先面向 `Windows + WSL + tmux`。

### 3.3 为什么不用现有 PTY 方案
- 现有实现会创建自己的 `node-pty` 子进程。
- 这会产生“飞书会话”和“你本地终端会话”两套上下文。
- 它不能稳定附着到现有 Windows Console。
- 因此它不满足“保留当前持续对话模式，再让飞书接入”的目标。

---

## 4. 核心组件

### 4.1 Relay
- 继续负责飞书 webhook、鉴权、审计、消息路由。
- 协议层保持简单，不感知底层是 PTY 还是 `tmux`。

### 4.2 Agent
- 不再负责创建长期 PTY 会话。
- 改为负责：
  - 检查目标 `tmux` session 是否存在
  - 把飞书文本写入指定 pane
  - 抓取 pane 输出增量
  - 回传状态和错误

### 4.3 Session Registry
- 维护工具名到共享会话目标的映射。
- 例如：
  - `cc -> session=bw-cc pane=0`
  - `cx -> session=bw-cx pane=0`
- 首期可用静态配置或内存配置实现。

### 4.4 Shared Runtime
- 真正承载上下文的是 WSL 内的 `tmux` session。
- `cc/cx` 都运行在其中。

---

## 5. 数据流

### 5.1 输入链路
- 飞书消息
- Relay 解析并路由
- Agent 根据当前工具找到目标 `tmux` pane
- Agent 执行 `tmux send-keys ... Enter`
- 输入进入共享会话内的 `cc/cx`

### 5.2 输出链路
- Agent 周期性或事件触发式执行 `tmux capture-pane`
- 与上次已发送内容做增量比对
- 仅把新增输出切块回传 Relay
- Relay 再回飞书

### 5.3 状态链路
- `/status` 时 Agent 查询：
  - `tmux` 是否存在
  - pane 是否存在
  - 近期是否有输出
- 然后回传给 Relay

---

## 6. 共享会话模型

### 6.1 工具与会话映射
- 方案 A：一个工具一个固定 session
  - `cc -> bw-cc`
  - `cx -> bw-cx`
- 首期采用方案 A，最简单稳定。

### 6.2 pane 约定
- 每个工具默认只桥接一个 pane。
- 首期不支持在飞书里切 pane。

### 6.3 cwd 语义
- `cwd` 不再由 Agent 动态创建 PTY 时注入。
- 共享会话的工作目录由你在 `tmux` 会话内自行维护。
- 飞书侧首期不保证 `/cwd` 生效；建议先降级为只读状态展示或禁用。

---

## 7. 协议与命令语义调整

### 7.1 保留的飞书命令
- `/cc`
- `/cx`
- `/status`

### 7.2 调整语义的命令
- `/reset`
  - 旧语义：重启 PTY
  - 新语义：重置飞书当前绑定状态，并重新探测共享会话

### 7.3 暂缓或降级的命令
- `/cwd`
  - 首期建议禁用，避免和共享会话模型冲突
- `/mode safe|yolo`
  - 首期建议降级为透传模式，不做自动确认
  - 因为共享会话中自动注入确认可能影响你本地终端的真实操作

---

## 8. 错误处理

### 8.1 必须显式回飞书的错误
- 未安装 WSL
- 未安装 `tmux`
- 找不到目标 session
- 找不到目标 pane
- 目标 session 存在但 pane 内未运行工具
- 输出抓取失败

### 8.2 静默失败禁令
- 任何输入发出后，如果无法完成注入或无法读取状态，必须回错误文本。
- 不允许“没回复也没日志”。

---

## 9. 测试策略

### 9.1 单元测试
- `tmux` 命令拼装
- Windows 路径到 WSL 路径转换
- 输出增量提取
- 错误消息映射

### 9.2 集成测试
- 在 WSL 中创建测试 `tmux` session
- 向 pane 注入文本
- 抓取输出并验证回传
- 本地终端输入后，飞书继续输入，确认上下文连续

### 9.3 手工验证
- 本地 attach 到 `bw-cx`
- 飞书发一句话
- 在本地看到同一句进入会话
- 再由飞书收到新输出

---

## 10. 风险与对策

### 10.1 输出增量抓取不准确
- 风险：重复回发或漏回发
- 对策：记录每个 pane 的最后游标或最后摘要，做稳定增量比较

### 10.2 本地与飞书同时输入
- 风险：内容交错
- 对策：首期接受共享终端天然串流特性，不做强事务隔离

### 10.3 `safe/yolo` 自动确认误伤本地操作
- 风险：飞书侧自动确认影响你本地终端中的真实确认
- 对策：首期关闭自动确认能力

---

## 11. 首期范围结论

首期只交付以下能力：
- WSL + `tmux` 共享会话桥接
- 飞书普通文本输入
- 飞书输出回显
- `/cc` `/cx` `/status`
- 明确错误回传

首期不交付：
- 接管任意现有 Windows PowerShell
- 飞书创建/删除/重命名会话
- `/cwd`
- 自动确认模式
